L3 VLAN Migration & NAT Gateway Setup

Implementation guide to segment K3s homelab traffic using Netgear GS716Tv3 routing capability and NUC asymmetric NAT.

⏱️ Migration Timeline & Planning Estimates

Phase Task Description Estimated Time
Phase 1 Pre-Migration Backup & Hardware Validation (WOL/LLDP check) 30 - 45 mins
Phase 2 Router DHCP Static IP Reservations 15 mins
Phase 3 Switch SVI & L3 IP Routing Configuration 20 mins
Phase 4 Switch Port Memberships & Trunk/Access Port Assignment 15 mins
Phase 5 Local Host Static IP Configuration (Linux Workers, Registry, Windows Workstation) 20 mins
Phase 6 NUC VLAN Netplan Tagging & IPTables Gateway Setup 25 mins
Phase 7 K3s Nodes systemd Configuration & Sequencing Restarts 45 mins
Phase 8 Comprehensive Validation & Diagnostics 20 mins
Total Maintenance Window (with testing & buffer) ~ 3 Hours 20 Mins

⚠️ Critical Deployment Safeguards (Verify Before Commencing)

1. Netgear Switch Save config: Netgear switches do not automatically save to NVRAM on application. After any settings adjustments, you must go to Maintenance > Save Config, tick the active slot, and click Apply.

2. K3s Node-IP Bindings: If node IPs change, Kubernetes certificates block communication. Service definitions on the master and workers must have --node-ip explicitly declared before rebooting.

3. Switch Gateway Asymmetry: The default route of the switch points to the NUC's VLAN 1 IP (acting as NAT forwarding engine), which masquerades packets to the Vodafone Router.

Phase 1: Pre-Migration Backups & Hardware Validation
1. Pre-Migration Backups

Perform full backups before altering switch ports or node routing configuration:

  • Switch Config: Go to Netgear GUI: Maintenance > Backup Manager > HTTP File Backup. Download config.cfg.
  • K3s Database: Run the manual backup script to save the cluster state to the QNAP NAS:
    sudo /usr/local/bin/k3s-backup-to-nas.sh
  • Current Node Mapping: Capture a snapshot of working IPs and MACs:
    kubectl get nodes -o wide > k3s_nodes_pre_migration.txt
2. LLDP Topology Check & Hardware Diagnostics

Deploy Link Layer Discovery Protocol to all cluster targets to map physical switch configurations:

pwsh -File .\WindowsLab\Install-LLDP.ps1

⚠️ BLOCKING HARDWARE TASK: Phase 1 Speed Mismatch

All K3s nodes responded via LLDP. However, speed warnings are registered on g10 (kubernetes7) and g11 (HP-2), which are bottlenecked at 100M.

Cable Diagnostic Steps:

  • Verify cable is Cat5e/Cat6 (must support 8 wires; 4-wire cables will auto-negotiate to 100M).
  • Inspect port pins on the Netgear switch and node NICs for debris or bent pins.
  • Verify speed negotiation: Go to switch GUI Monitoring > Ports and check if the port is set to Auto. Do not force 1000M; if the link is bad, forcing it will disable the port entirely.
This is a blocking issue that must be resolved (must negotiate 1000M/Full) before proceeding to Phase 2 of the migration to ensure storage and network health.
2
Phase 2: Router DHCP Reservations

⚠️ ROUTER DHCP LIMITATIONS & IP ASSIGNMENT

Important: Consumer ISP routers (like the Vodafone Router) are locked down and do not support serving multiple DHCP scopes/pools for custom VLANs, nor do they allow assigning multiple static IPs to a single MAC address.

To work around this limitation:
  • Configure a **static DHCP reservation** on the Vodafone Router only for the NUC's VLAN 1 interface (192.168.0.21).
  • Statically configure all other IPs directly on the operating systems of the respective hosts (NUC VLAN 20 interface, workers, registry, and HP-2 workstation) rather than relying on DHCP.

Configure the IP assignment mapping as follows:

Device IP Assignment Type Subnet / VLAN Assigned IP
NUC Master (VLAN 1) DHCP Static Reservation (Vodafone Router) VLAN 1 (Flat) 192.168.0.21
NUC Master (VLAN 20) Static IP (Configured locally on NUC Netplan) VLAN 20 (K3s) 192.168.20.21
HP-1 Registry Static IP (Configured locally on Host) VLAN 20 (K3s) 192.168.20.236
HP-2 Workstation Static IP (Configured locally on Host) VLAN 30 (Trusted) 192.168.30.122

Verify the IP planning layout in the mapping table. Local static IP configuration procedures are detailed in Phase 5 of the guide.

⚠️ Static Route Verification Not Found

The Vodafone router routing panel is locked down. Proceed with NUC Next-Hop NAT translation architecture (Option A.2).

Phase 3: Switch L3 Interface Configuration

Configure Switched Virtual Interfaces (SVIs) and Default Routing on the Netgear GS716Tv3 switch Web Interface:

  1. Navigate to Routing > IP > IP Configuration, set Routing Mode to Enable, and click apply.
  2. Go to Routing > IP > IP Interface Configuration and create routing interface addresses:
    • VLAN 1: 192.168.0.99 (Mask: 255.255.255.0)
    • VLAN 10 (MGMT): 192.168.10.99
    • VLAN 20 (K3s): 192.168.20.99
    • VLAN 30 (Trusted): 192.168.30.99
  3. Set default routing under Routing > IP > Static Route Configuration:
    Destination: 0.0.0.0 Subnet Mask: 0.0.0.0 Next Hop Gateway: 192.168.0.21 (NUC VLAN 1 Interface - NOT Vodafone Router)
  4. Skip DHCP Relay: Since the Vodafone Router cannot serve multiple DHCP scopes, DHCP Relay is not used. Ensure all nodes in VLAN 20 and 30 are configured with static IPs locally.

⚠️ Configuration Persistence Requirement

Go to Maintenance > Save Config, check the checkbox and click Apply, or your configuration will vanish upon next powercycle.

Phase 4: Port Memberships & Trunk Setup

Assign VLAN IDs and adjust trunking configuration on the switch ports:

  1. Navigate to Switching > VLAN > Basic > VLAN Configuration and create VLAN IDs 10, 20, 30.
  2. Set Port g8 (NUC Link) as trunk:
    • VLAN 1: Set to Untagged (U), PVID = 1.
    • VLAN 20: Set to Tagged (T).
  3. Set K3s nodes ports (g2-g6, g10, g12, g15) to VLAN 20:
    • PVID Configuration: Set PVID to 20.
    • VLAN Membership: Set to Untagged (U) on VLAN 20.
  4. Set HP-2 (g11) to VLAN 30:
    • PVID Configuration: Set PVID to 30.
    • VLAN Membership: Set to Untagged (U) on VLAN 30.

Phase 5: Local Host Static IP Configuration

Configure the static IP addresses locally on each device to match their respective VLAN segments:

1. Linux Worker Nodes (VLAN 20)

For each worker node (kubernetes1 to kubernetes8), edit the Netplan configuration file (typically /etc/netplan/00-installer-config.yaml or 50-cloud-init.yaml):

sudo nano /etc/netplan/00-installer-config.yaml
network: version: 2 ethernets: eth0: # Note: verify interface name (e.g. eth0, enp3s0) using 'ip link' addresses: - 192.168.20.22/24 # Unique IP per worker (e.g. .22 for kubernetes1, .23 for kubernetes2, etc.) routes: - to: 0.0.0.0/0 via: 192.168.20.99 # Switch SVI IP as gateway nameservers: addresses: [192.168.20.99, 8.8.8.8]

Apply the changes:

sudo netplan apply
2. HP-1 Registry Node (VLAN 20)

Edit the Netplan config on HP-1 to configure the static IP 192.168.20.236:

sudo nano /etc/netplan/00-installer-config.yaml
network: version: 2 ethernets: eth0: # Note: verify interface name (e.g. eth0, enp3s0) using 'ip link' addresses: - 192.168.20.236/24 routes: - to: 0.0.0.0/0 via: 192.168.20.99 # Switch SVI IP as gateway nameservers: addresses: [192.168.20.99, 8.8.8.8]

Apply the changes:

sudo netplan apply
3. HP-2 Windows Workstation (VLAN 30)

Configure the static IP 192.168.30.122 on HP-2 using either the Windows GUI or PowerShell:

  • GUI Method:
    1. Navigate to Network & Internet > Advanced network settings > More network adapter options (or run ncpa.cpl).
    2. Right-click your adapter (e.g., Ethernet) and select Properties.
    3. Double-click Internet Protocol Version 4 (TCP/IPv4).
    4. Select Use the following IP address:
      • IP address: 192.168.30.122
      • Subnet mask: 255.255.255.0
      • Default gateway: 192.168.30.99 (Switch SVI)
    5. Select Use the following DNS server addresses:
      • Preferred DNS server: 192.168.30.99
      • Alternate DNS server: 8.8.8.8 (or 1.1.1.1)
  • PowerShell Method (Run as Administrator):
    New-NetIPAddress -InterfaceAlias "Ethernet" -IPAddress "192.168.30.122" -PrefixLength 24 -DefaultGateway "192.168.30.99" Set-DnsClientServerAddress -InterfaceAlias "Ethernet" -ServerAddresses ("192.168.30.99", "8.8.8.8")

Phase 6: NUC VLAN Tagging & IPTables Gateway Setup

Configure Netplan on the NUC to receive tagged VLAN 20 traffic and act as a NAT masquerade engine.

1. Edit /etc/netplan/01-netcfg.yaml (or equivalent netplan config):

network: version: 2 ethernets: enp3s0: dhcp4: true routes: - to: 192.168.10.0/24 via: 192.168.0.99 - to: 192.168.30.0/24 via: 192.168.0.99 vlans: vlan20: id: 20 link: enp3s0 addresses: - 192.168.20.21/24

2. Apply netplan config:

sudo netplan apply

3. Enable IP Forwarding and NAT rules:

# Enable forwarding instantly sudo sysctl -w net.ipv4.ip_forward=1 echo "net.ipv4.ip_forward=1" | sudo tee -a /etc/sysctl.conf # Add NAT masquerade rules sudo iptables -t nat -A POSTROUTING -o enp3s0 -j MASQUERADE # Allow forwarding for all internal private subnets (VLAN 10, 20, 30) out to internet sudo iptables -A FORWARD -s 192.168.0.0/16 -o enp3s0 -j ACCEPT sudo iptables -A FORWARD -m conntrack --ctstate RELATED,ESTABLISHED -j ACCEPT # Save iptables rules permanently sudo apt-get install iptables-persistent -y sudo iptables-save | sudo tee /etc/iptables/rules.v4

Phase 7: K3s Node Settings Adjustment & Sequencing Restarts

⚠️ K3S RESTART SAFEGUARDS & ROLLOUT RISK

Do NOT restart all K3s master and worker services simultaneously. Doing so will drop cluster quorum, cause Longhorn storage volume disconnects, and lead to database corruption.

Recommended Sequence:

  1. Configure Netplan and update systemd on the Master Node (NUC). Restart the k3s master service first.
  2. Verify the master is fully operational and the API server is responsive (kubectl get nodes should run).
  3. Update and restart the worker node services one by one, starting with the database workers. Wait for each worker to reach Ready status before moving to the next.

Adjust systemd configuration on K3s nodes to bind to the new VLAN 20 IPs.

1. Master Node NUC configuration (/etc/systemd/system/k3s.service):

# Inside ExecStart command, append node-ip and advertise-address variables ExecStart=/usr/local/bin/k3s server \ --node-ip=192.168.20.21 \ --advertise-address=192.168.20.21 \ --flannel-backend=wireguard-native

2. Worker Node configuration (/etc/systemd/system/k3s-agent.service.env on e.g., kubernetes1):

K3S_URL="https://192.168.20.21:6443" K3S_TOKEN="K101..." K3S_AGENT_ARGS="--node-ip=192.168.20.22"

3. Reload & Restart services (Sequenced):

# Run on NUC Master first, then sequentially on each worker sudo systemctl daemon-reload sudo systemctl restart k3s # or k3s-agent for workers

Phase 9: Troubleshooting & Debugging

Use these diagnostic guides if you experience common issues during migration:

1. Lost Connectivity to Switch

If you lose access to the Netgear switch GUI after enabling routing:

  • Temporary Fix: Connect your workstation directly to port g16 (which should remain in default VLAN 1 untagged) and statically configure your workstation IP to 192.168.0.100. Navigate to 192.168.0.99.
  • Full Factory Reset: Press and hold the physical Reset button on the switch front panel for 10 seconds to restore factory defaults. Restore from the pre-migration config.cfg backup file.
2. Netplan / IP Forwarding Issues

If the NUC is not forwarding packets for other subnets, verify that the Linux kernel has IP forwarding enabled and netfilter modules loaded:

# Verify IP Forwarding is active (should output 1) cat /proc/sys/net/ipv4/ip_forward # Check if iptables-persistent is active at boot systemctl status iptables-persistent # Check active NAT forwarding rules sudo iptables -t nat -L -v -n
3. K3s Certificate / TLS Errors

If K3s agents display certificate validation errors or cannot join the control plane:

  • Verify that the Master node's k3s.service configuration has --node-ip=192.168.20.21 and --advertise-address=192.168.20.21 explicitly set.
  • Verify that you can reach the API port 6443 on the master from workers:
    nc -zv 192.168.20.21 6443
  • If certificate errors persist, you may need to force-regenerate agent tokens or remove cached certificates on workers under /var/lib/rancher/k3s/agent/.

Phase 10: Disaster Rollback Plan

🚨 DISASTER ROLLBACK PROCEDURE

If the cluster fails to converge or network routing is unstable, follow these steps to revert the entire lab back to the original flat configuration (192.168.0.0/24).
Step 1: Revert K3s Node Settings

Revert K3s settings on all nodes to bind back to the flat IPs:

  1. On the NUC Master, restore /etc/systemd/system/k3s.service (remove --node-ip and --advertise-address flags).
  2. On all Workers, restore /etc/systemd/system/k3s-agent.service.env to point back to the master at https://192.168.0.21:6443 and set worker IPs back to 192.168.0.x.
  3. Run:
    sudo systemctl daemon-reload
Step 2: Revert Netplan on the NUC

Restore Netplan to its flat configuration by removing the VLAN 20 configuration block in /etc/netplan/01-netcfg.yaml, leaving only the primary network interface on DHCP or flat IP configuration. Then apply it:

sudo netplan apply
Step 3: Revert Port Memberships and SVI on Switch

Navigate to the Netgear Switch Web Management GUI:

  1. Go to Switching > VLAN > Advanced > VLAN Membership. Set all ports (g1 to g16) back to Untagged (U) on VLAN 1.
  2. Set the PVID for all ports back to 1.
  3. Go to Routing > IP > IP Interface Configuration and delete SVI interfaces for VLAN 10, 20, and 30.
  4. Go to Routing > IP > IP Configuration, set Routing Mode to Disable, and click apply.
  5. Go to Maintenance > Save Config, check the checkbox and click Apply to commit rollback permanently.
Step 4: Verify Flat Connectivity

Run a ping trace from your workstation to confirm that all nodes are reachable directly at their 192.168.0.x addresses on the flat network.
