☸️ Kubernetes Security & Best Practices Analysis Report

Date: June 09, 2026 | Auditor: Antigravity (using kubernetes-specialist skill) | Target Directory: kubernetes-manifests/base/

📋 Executive Summary

An audit of the core homelab manifests (iiqstack) was conducted using the Kubernetes Specialist standards. While these manifests function correctly for a local developer environment, they contain several production anti-patterns and security vulnerabilities:

  • Hardcoded Credentials: Plaintext passwords are exposed directly in environment variables.
  • Lack of Resource Limits: Containers run without CPU/Memory limits, leaving the nodes vulnerable to starvation or out-of-memory (OOM) crashes.
  • Over-privileged Security Contexts: Database init containers and workloads run as root with privilege escalation enabled.
  • Missing Probes: Key components (e.g., MSSQL database) lack liveness or readiness health checks.
  • Default Namespaces: All base manifests are deployed to the default namespace instead of logical isolation.

🔍 Detailed Audits

1. IdentityIQ Workload (iiq-deployment.yaml)

⚠️
WARNING: This deployment suffers from major security exposures due to credential leakage and root filesystem mutability.
  • Hardcoded DB Credentials:
    - name: MSSQL_PASS
  value: Id3nt!tyIQ2026
  • No Resource Limits: If Java garbage collection or memory leaks occur, the container can consume all node RAM, causing the K3s host to crash.
  • Missing Liveness Probe: If the Tomcat server locks up, Kubernetes won't auto-restart the container because only readinessProbe and startupProbe are configured.

2. MSSQL Database Workload (db-mssql-deployment.yaml)

🚫
CAUTION: Running stateful databases via standard Deployment instead of StatefulSet risks database corruption during rescheduling.
  • Deployment vs. StatefulSet: Deployments do not guarantee a single instance or stable network storage binding during node failures. This can cause split-brain data corruption when scaling or moving the pod.
  • Over-Privileged Permission Fix:
    chmod 777 /var/opt/mssql
    Setting directory permissions to 777 allows any container running on the host or inside the pod to read/write database files.

🛠️ Refactoring Recommendations

To align your manifests with the Kubernetes Specialist standard, implement the following changes:

A. Extract Credentials into a Kubernetes Secret

Create a secret named db-secrets.yaml:

apiVersion: v1
kind: Secret
metadata:
  name: db-secrets
  namespace: iiqstack
type: Opaque
stringData:
  mssql-pass: "Id3nt!tyIQ2026"

And mount it securely in the workload manifests:

- name: MSSQL_PASS
  valueFrom:
    secretKeyRef:
      name: db-secrets
      key: mssql-pass

B. Add Resource Specifications and Probes

Apply constraints and full health checks to iiq-deployment.yaml:

resources:
  requests:
    cpu: "250m"
    memory: "1Gi"
  limits:
    cpu: "1000m"
    memory: "2Gi"
livenessProbe:
  httpGet:
    path: /identityiq
    port: 8080
  initialDelaySeconds: 90
  periodSeconds: 20

C. Restrict Pod Security Contexts

Update the security context to run containers as non-root and restrict permissions:

securityContext:
  runAsNonRoot: true
  runAsUser: 10001
  fsGroup: 2000
